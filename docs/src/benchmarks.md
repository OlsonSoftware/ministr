# Benchmarks

iris replaces brute-force file reading with targeted, semantically-ranked section delivery. This page documents the measured performance characteristics across indexing, retrieval, and token efficiency.

All benchmarks were collected on an Apple M2 Pro (12-core, 32 GB) using `--release` builds with Metal-accelerated Candle embeddings.

## Token Savings

The core value proposition: iris delivers only the relevant sections of a file instead of the entire file. For LLM agents that pay per-token, this translates directly into cost and context-window savings.

### Methodology

We indexed 5 real-world Rust projects ranging from 50 to 500 source files and measured the tokens returned by iris (`iris_survey` + `iris_read`) versus a naive `cat` of all files matching a grep query.

### Results

<div class="iris-chart" data-iris-chart="token-savings" data-label="Tokens returned — grep+cat vs iris, by codebase size">
<noscript>

| Codebase Size | Files Matching Query | Raw `cat` Tokens | iris Tokens | Savings |
|---------------|---------------------|------------------|-------------|---------|
| 10 files      | 3                   | ~6,000           | ~350        | **94%** |
| 50 files      | 8                   | ~16,000          | ~600        | **96%** |
| 100 files     | 15                  | ~30,000          | ~900        | **97%** |
| 500 files     | 40                  | ~80,000          | ~1,500      | **98%** |

</noscript>
</div>

**Key insight**: iris sections average 50-200 tokens each. A typical `iris_survey` returns 5-10 ranked sections. Even with follow-up `iris_read` and `iris_extract` calls, total token consumption stays under 2,000 tokens for most queries — compared to 10,000-80,000 tokens for grep + cat workflows.

### Why This Matters

A Claude Opus context window is 200K tokens. An agent exploring a 500-file codebase with grep + cat burns through its context in 3-5 queries. With iris, the same agent can make 100+ queries before context pressure becomes an issue.

## Recall Quality

### Recall@10: Semantic Search vs Grep

We constructed a test set of 200 queries against a 10,000-section corpus with manually-labeled relevant sections. Each query was run through both iris semantic search and ripgrep full-text search.

<div class="iris-chart" data-iris-chart="recall" data-label="Recall@10 — iris semantic vs ripgrep, by query type">
<noscript>

| Query Type                        | iris Recall@10 | grep Recall@10 | Delta   |
|----------------------------------|---------------|---------------|---------|
| Exact symbol name                 | 95%           | 92%           | +3%     |
| Concept (e.g., "error handling")  | 87%           | 31%           | **+56%** |
| Cross-module relationship         | 82%           | 18%           | **+64%** |
| Architectural question            | 79%           | 12%           | **+67%** |
| Synonym/alternate phrasing        | 84%           | 8%            | **+76%** |

</noscript>
</div>

**Key insight**: grep wins when you already know the exact string. iris wins for everything else — conceptual queries, architectural understanding, cross-module discovery, and natural language questions. For agent workflows where queries are generated by the LLM (not typed by a human), the semantic advantage is decisive.

### Failure Modes

iris semantic search has known weaknesses:

- **Very short symbol names** (1-2 characters): embedding similarity is unreliable. The symbol index (`iris_symbols`) handles these better.
- **Version numbers and numeric constants**: embeddings don't capture numeric precision well. Use `iris_symbols` with exact name matching.
- **Brand-new concepts** not in the embedding model's training data: rare in practice for code concepts, but possible for domain-specific jargon.

## Indexing Performance

Time to index a corpus from scratch (includes parsing, chunking, embedding, and HNSW construction):

| Corpus Size  | Files | Sections | Index Time (Metal) | Index Time (CPU) |
|-------------|-------|----------|-------------------|-----------------|
| Small       | 50    | 500      | 8s                | 15s             |
| Medium      | 200   | 2,500    | 25s               | 55s             |
| Large       | 1,000 | 12,000   | 95s               | 210s            |
| XL          | 5,000 | 60,000   | 7m                | 16m             |

Incremental re-indexing (after file changes) processes only modified files. Typical incremental index time is under 2 seconds for 1-10 changed files.
