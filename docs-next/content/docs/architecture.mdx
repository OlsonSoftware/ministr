## Workspace Structure

ministr is organized as a Cargo workspace with six crates:

```text
ministr-core/          — domain logic, no transport dependencies
ministr-api/           — shared request/response types for daemon ↔ MCP/CLI communication
ministr-daemon/        — HTTP API over Unix domain socket, depends on ministr-core + ministr-api
ministr-mcp/           — MCP server adapter, depends on ministr-core + ministr-api + rmcp
ministr-cli/           — binary entry point, depends on ministr-mcp
ministr-app/src-tauri/ — Tauri v2 desktop app, depends on ministr-core + ministr-api + ministr-daemon
```

**ministr-core** contains all business logic: parsing, indexing, embedding, search, session management, prefetching, and usage accounting. It has no knowledge of MCP or any transport protocol.

**ministr-api** defines the shared request/response types and the `DaemonClient` for communicating with the ministr daemon over a Unix domain socket (`~/.ministr/ministrd.sock`). It has no dependency on ministr-core — it is pure types.

**ministr-daemon** is the long-running background service that owns the heavy resources (ONNX model, HNSW indexes, SQLite). It exposes an HTTP API over UDS for corpus management and querying.

**ministr-mcp** adapts ministr-core's service layer to the MCP protocol using the `rmcp` crate. It implements tool handlers, request routing, and response formatting. Ships in two flavours: the standalone `MinistrServer` that owns its own storage/embedder/index, and a `ProxyServer` that delegates to the daemon over UDS so a second client doesn't have to load a second ONNX model into RAM.

**ministr-cli** is the binary entry point. It parses CLI arguments, loads configuration, and starts the MCP server over stdio transport.

**ministr-app** is the Tauri v2 desktop application that wraps the daemon with a GUI dashboard and system tray icon.

## The Mechanisms

ministr combines a few mechanisms. Hybrid search and the retrieval pipeline are how it understands code; the session-tracking mechanisms keep delivery lean so the agent spends its window on the work, not on re-reading.

### 1. Session Shadow
A per-session record of every chunk of code or docs ministr has already sent the agent. Lets ministr skip re-sending, return a pointer instead of the same text, or compute a delta when something changes.

### 2. Delta Delivery
When the agent re-reads a file that has changed since it last saw it, ministr sends only the changed lines — not the whole file again. Built on top of the Session Shadow.

### 3. Predictive Prefetch
After every read, ministr guesses what the agent will look up next (sequentially, by topic, or by document structure) and pre-computes those results. The next call lands on a pre-warmed result instead of running a fresh search.

### 4. Lean Delivery
ministr keeps an advisory estimate of how many tokens it has delivered and can return compact claim-level extracts instead of full sections when asked. This is internal accounting, not a signal the agent should treat as "low on context."

### 5. Coherence Watcher
A file watcher that notices when files change on disk after the agent has read them, marks the delivered content stale, and surfaces alerts on the next tool call so the agent doesn't reason off rotted context.

### 6. Hybrid Search
Semantic ("meaning") search and keyword search run together, with their rankings fused — so the agent finds the right function whether it searches by what the function does or what it's called.

### Retrieval Pipeline (how the search above works)

The first five mechanisms sit on top of a retrieval pipeline that powers `ministr_survey` and `ministr_read`:

- **Multi-resolution indexing** — every document is indexed at three levels at once: summary, section, and atomic claim (a one-sentence fact pulled from a section). The agent can pick the level it needs.
- **Progressive disclosure** — the tool surface (`survey` → `read` → `extract` → `related`) lets the agent zoom in only as far as it actually needs to, instead of pulling whole files up-front.

For the parser, embedding, and index internals, see the [architecture deep dive](/docs/architecture-deep-dive).

## On-Disk Layout

```text
~/.ministr/
├── config.toml                       # Global configuration
├── corpora.json                      # Daemon's registered-corpus manifest
├── ministrd.sock                     # Daemon socket (UDS on macOS/Linux; named pipe on Windows)
├── ministrd.pid                      # Daemon PID file
└── corpora/
    └── <corpus-id>/                  # Stable hash derived from corpus paths
        ├── meta.toml                 # Per-corpus configuration (name, model, watch flag, ...)
        ├── content.db                # SQLite: sections, claims, summaries, file hashes
        ├── sessions/                 # Persisted session shadows
        └── index/
            ├── ministr_hnsw.hnsw.data  # HNSW vector dump
            ├── ministr_hnsw.hnsw.graph
            └── id_map.json           # Section ID ↔ vector slot mapping
```

## Key Dependencies

| Purpose | Crate |
|---|---|
| Embeddings | `fastembed` (ONNX-based, local) |
| Vector search | HNSW index (in-memory, memory-mapped) |
| Storage | `rusqlite` (SQLite) |
| MCP protocol | `rmcp` |
| Document parsing | `comrak` (Markdown), `scraper` (HTML), `pdf-extract` (PDF) |
| Async runtime | `tokio` |
| Observability | `tracing` |
| File watching | `notify` |
