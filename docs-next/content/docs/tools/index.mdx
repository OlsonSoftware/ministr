ministr exposes MCP tools that give the agent explicit control over context retrieval. The tools cover document search, code navigation, cross-language bridges, budget management, and multi-source corpus ingestion.

## Search & retrieval

| Tool | Purpose | Typical Token Cost |
|---|---|---|
| [`ministr_survey`](/docs/tools/survey) | Semantic search across docs and code | 100-300 |
| [`ministr_read`](/docs/tools/read) | Read a section in full | 200-2000 |
| [`ministr_extract`](/docs/tools/extract) | Pull specific claims | 50-500 |
| [`ministr_related`](/docs/tools/related) | Follow claim relationships | 50-300 |
| [`ministr_toc`](/docs/tools/toc) | Structural overview of the corpus | 100-500 |

## Response envelope

Every MCP tool returns a `ToolResponse<T>` envelope. The tool-specific payload lives under `result`; everything else is wrapper metadata that the agent can use to track session state without re-querying.

```json
{
  "budget_status": {
    "tokens_used": 34500,
    "tokens_remaining": 65500,
    "pressure_level": "normal",
    "utilization": 0.345
  },
  "coherence_alerts": [ /* present only when underlying content has changed */ ],
  "indexing_in_progress": true,
  "indexing_message": "Checking 12/42 files",
  "eviction_recommendations": [ /* present when budget pressure is elevated or critical */ ],
  "result": {
    /* tool-specific payload — shape varies per tool */
  }
}
```

| Wrapper field | Always present? | Description |
|---|---|---|
| `budget_status` | ✅ always | Current token-budget snapshot (`tokens_used`, `tokens_remaining`, `pressure_level`, `utilization`) |
| `coherence_alerts` | only when non-empty | Notifications for sections whose underlying content changed since last delivery |
| `indexing_in_progress` | only when `true` | Background corpus ingestion is still running |
| `indexing_message` | only when present | Human-readable ingestion status (e.g. `"Checking 12/42 files"`) |
| `eviction_recommendations` | only when budget is elevated/critical | Proactive eviction candidates — same shape as the `eviction_candidates` field in [`ministr_budget`](/docs/tools/budget) |
| `result` | ✅ always | Tool-specific payload — see each tool's reference page for its shape |

The per-tool reference pages below show the full envelope in their `Response` examples for clarity. Field tables describe the `result` payload (shape varies per tool) and reference this envelope for the wrapper fields.

## Typical Workflow

```text
survey → read → extract → related
  ↓                          ↓
budget ← compress ← evicted
```

1. **Survey** to orient — find relevant areas of the corpus
2. **Read** specific sections identified by the survey
3. **Extract** claims for surgical precision
4. **Related** to follow reasoning chains
5. **Budget** to check pressure and get eviction recommendations
6. **Compress** sections before evicting them
7. **Evicted** to tell ministr what was dropped (improves tracking accuracy)
