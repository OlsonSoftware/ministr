---
title: Session memory
description: How ministr remembers what it has already shown an agent, so it skips re-sends and ships only what changed.
---

ministr keeps a per-session memory of everything it has shown the agent. That memory is what makes delivery lean — it is a quiet efficiency layer beneath the code intelligence, not the headline.

## What it enables

- **Skip repeats.** If the agent searches for something ministr already provided, it returns a pointer instead of the same text again — the agent gets only what's new.
- **Ship deltas.** If a section was sent earlier and the file has since changed, ministr sends just the changed lines, not the whole thing again. The agent sees exactly what moved.
- **Re-supply what was dropped.** ministr keeps a rough sense of how much it has sent relative to the agent's window, so if the agent asks again for context it likely pushed out, ministr can re-deliver it.

## Why a rough estimate is enough

ministr can't see the agent's real context window — MCP doesn't expose it — so it works from an estimate, corrected as it goes: if the agent re-requests something it was already given, ministr treats that as a signal the content was dropped and re-supplies it. The estimate doesn't need to be exact. Even a loose one prevents the most wasteful failure mode — re-sending identical context every turn — and it self-corrects with use.

This memory survives across restarts, so reconnecting to a session doesn't start the agent over from scratch.
