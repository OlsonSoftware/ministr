title: Session Shadow
description: A per-session ledger of every chunk of context ministr has delivered to the agent — so it skips re-sending what the agent already has and ships only what changed.
---

The session shadow is a per-session ledger of everything ministr has delivered to the agent. It tracks exactly what context has been sent, enabling three capabilities that no existing retrieval system provides.

## What It Tracks

For every piece of content delivered, the shadow records:

- **Content ID** — which summary, section, or claim was sent
- **Resolution** — at what level (summary, section, or claim)
- **Token count** — how many tokens it consumed
- **Turn number** — when in the conversation it was delivered
- **Content hash** — for detecting changes to the underlying document

## Capabilities

### Skip Repeats

When the agent searches for a topic that ministr already provided context for, the shadow filters out previously-delivered content. The agent receives only new information, not copies of what it already has.

### Delta Updates

If a section was delivered earlier but the underlying document has changed, ministr detects the hash mismatch and returns a line-level delta instead of re-delivering the full text. The agent sees exactly what changed.

### Drop Estimation

Based on the configured window size and cumulative token delivery, ministr makes a rough estimate of what the agent has likely dropped from its window, so it can re-deliver critical context if the agent asks for it again.

## Window Estimation

ministr does not have direct access to the agent's actual context window (MCP does not expose this). Instead, it maintains an estimate based on:

1. **Cumulative token count** — when delivery exceeds the configured window size, older items are assumed dropped (FIFO by default, or FSRS-based when a `MemoryTracker` is wired in and sessions record with retrievability scores)
2. **Fault-based correction** — if the agent re-requests content ministr already delivered, this is treated as a signal that it was dropped. The window estimate is corrected and the content is re-delivered.
3. **Explicit signals** — the agent can call `ministr_evicted` to tell ministr what it dropped, improving accuracy

The estimate does not need to be perfect. Even a rough approximation prevents the most wasteful failure mode (re-delivering identical context every turn) while fault-based correction converges on accuracy over time.

## Persistence

Session shadows are persisted to SQLite so sessions can survive process restarts. When ministr starts with an existing session ID, the shadow is restored from storage.
