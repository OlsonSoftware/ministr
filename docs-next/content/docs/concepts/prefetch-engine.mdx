The **predictive prefetch** engine predicts what the agent will ask for next and pre-computes it, so follow-up calls return instantly instead of running a fresh search.

## Why Prefetch Matters

Every tool call round-trip costs latency, tokens, and money. If ministr can guess what the agent will ask for next and have it ready, the interaction feels instantaneous — and the next response can skip an embed-and-search cycle entirely.

No LLM is in the loop for prediction itself. If guessing what to prefetch took 500ms of inference, it would defeat the purpose.

## The Six Strategies

Prefetch runs at two distinct moments in the agent's workflow. Four strategies fire after every `ministr_read` (sequential, structural, topical, cross-session); two fire after every `ministr_survey` (survey-expand, agent-plan).

### After a read

**Sequential locality.** When the agent reads section N of a document, pre-warm section N+1 (and a short window beyond). This mirrors hardware prefetch streaming.

**Structural locality.** Pre-warm sibling sections at the same depth around the current read position, so "read through a doc" and "look at the next thing in the outline" both serve warm.

**Topical locality.** Keep a running "topic vector" — a weighted average of the embedding vectors of the last K sections the agent accessed — and query HNSW for its nearest un-read neighbours. Analogous to stride-based prefetching, generalised to a high-dimensional embedding space.

**Cross-session analytics.** Pull sections that, across previous sessions, have often been read together with the one the agent just read. Only runs in ministr's default single-process mode today; the daemon-proxy path has this scaffolded but not yet wired.

### After a survey

**Survey expand.** When a survey returns claim-level hits, pre-warm each hit's parent section — so the typical `survey → read` follow-up is already in cache.

**Agent plan (intent-based).** Cheaply analyse the query plus the top survey hits to infer what the agent is likely to ask for next, and pre-warm a small capped list. Capped to avoid wasted prefetching.

## Cache

Pre-warmed results live in an in-memory LRU cache (default size: 128 items). Each entry holds the pre-computed text, token count, relevance score, and a tag recording which strategy warmed it — so hit-rate metrics break down per strategy and you can see which predictions actually pay off.

## Configuration

```toml
[prefetch]
enabled = true                   # Toggle prefetching on/off
cache_size = 128                 # Max items in the prefetch cache
topic_window = 5                 # Recent sections used for the topic vector
alpha = 0.3                      # EMA weight on the most recent section (optional)
adaptive_alpha = true            # Auto-tune alpha when the topic stabilises (optional)
speculative = false              # Run prefetch proactively during agent idle (optional)
max_candidates_per_strategy = 5  # Cap per strategy per cycle (optional)
```

Prefetching can be disabled entirely with `enabled = false`. `cache_size` controls memory usage (each cached item holds a full section or claim). The optional fields are fine-tuning knobs — most users never touch them.
