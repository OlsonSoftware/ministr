---
title: Predictive prefetch
description: How ministr has the agent's likely next request ready before it asks — predicted, not LLM-generated.
---

Predictive prefetch guesses what the agent will ask for next and prepares it in advance, so the follow-up call returns immediately instead of running a fresh search.

## Why it matters

Every tool round-trip costs latency and tokens. If ministr can anticipate the next request and have the answer ready, the interaction feels instant and the next response skips a full search cycle. Prediction itself uses no LLM — if guessing took a model call, it would defeat the purpose.

## What it predicts from

Prefetch reasons from where the agent already is, at two moments:

- **After a read** — the agent is usually moving through related code: the next part of the same file, neighbouring sections, work on the same topic, and things that, in past sessions, were frequently read alongside what was just read.
- **After a search** — the natural next step is to open the strongest results, so ministr pre-warms those, plus a small, capped set of sections the query implies the agent is heading toward.

It stays conservative: predictions are capped so prefetch never does more work than it saves.

## Configuration

A few `.ministr.toml` knobs; the defaults are sensible and most users never touch them.

```toml
[prefetch]
enabled = true                   # Toggle prefetching on/off
cache_size = 128                 # Max pre-warmed items kept in memory
topic_window = 5                 # Recent sections used to infer the current topic
max_candidates_per_strategy = 5  # Cap on how much each prediction prepares
```

Set `enabled = false` to turn it off entirely. See [Configuration](/docs/configuration) for the full list.
