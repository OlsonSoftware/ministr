The app is not a dashboard. It is a mirror of your agent's knowledge, built to answer three plain questions: *is my AI up to date?*, *what exactly does it see?*, and *what did it actually do?* It runs in your system tray; the ministr engine itself is a background daemon that keeps your indexes warm even when the app window is closed.

## The trust vocabulary

Every file and project is described with four states. The glyphs differ in **shape**, not just color, so the state is readable without color vision:

| Mark | State | Meaning |
|---|---|---|
| ✓ | up to date | what your AI reads matches your working tree |
| ⚠ | behind your changes | your AI sees an older version |
| ✗ | hidden from your AI | not in the index |
| ⟳ | updating | being brought up to date right now |

These verdicts are honest by construction: every one is produced by **hashing the working tree** — file timestamps are never trusted. See [freshness](/docs/concepts/freshness).

## Home — the trust panel

One row per project, worst first; healthy projects stay quiet. A typical row reads "Your AI is 3 files behind — it may answer from old code," with a **Catch up** button that triggers a reindex. Rows show "N agents connected" when sessions are live. If ministr isn't running, the app says so and reconnects automatically — it never shows stale data as fresh (a "showing the last good view; reconnecting…" note appears instead).

## The project mirror

Click a project to see your file tree **as a trust display**: every file carries its mark, directories roll up to their worst child, and healthy subtrees stay collapsed. The subtitle is literal: *what your AI sees.*

Click a file to drill in. The drill-in shows the **stored indexed sections** — the exact text retrieval serves to your agent, never re-derived for display. If the file is behind, a banner offers a toggle between **"As your AI sees it"** and **"My current file"** so you can see precisely what's outdated.

A presence line at the bottom tells you when your AI is actively looking: live within the last 15 seconds (with a breathing dot), "last looked N minutes ago" within 10 minutes, and nothing at all beyond that — no event, no line.

An **advanced** disclosure in the rail exposes internals (corpus id, section counts) and an expert configuration panel where you can change the embedding model for this project — the one GUI knob, and changing it re-reads the whole project. It writes to the same `.ministr.toml` the CLI reads, so the GUI and the file can never disagree.

## The proof feed

"What ministr did" opens a receipts list: one plain-English line per recorded event, newest first. Tool calls become sentences — *your AI searched "auth token refresh"*, *your AI read daemon.rs*, *your AI skimmed the project layout*. Outcome receipts state read→edit joins: ✓ *you changed X — the first file your AI read*, or ⚠ *you changed X — your AI had read it earlier*. A bottom line aggregates counts only: searches, reads, answers from memory, files-read-then-edited.

An expert disclosure shows the raw ledger: timestamp, tool, summary, cache hit, token delta.

## The connect flow

Adding a project walks three beats — pick a folder, watch ministr read it, copy the one-line client config — and then waits for the **handshake**: the screen lights up with "Your AI just saw your code" only when a genuine first tool call arrives. It is never simulated.

## Behavior notes

- **Theme** follows your OS by default; a System / Light / Dark picker lives on Home.
- **Keyboard**: Escape walks back one level everywhere; the file tree supports full arrow-key navigation.
- **Daemon lifecycle**: the app spawns (or attaches to) the shared `ministr` daemon at boot. The daemon deliberately outlives the app, so closing the window doesn't cold-start your next session.
