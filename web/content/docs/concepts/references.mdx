Once ministr knows the [symbols](/docs/concepts/symbols) in a codebase, it connects them: which functions call which, which types implement which traits or interfaces, which modules import which. That web of connections is the **reference graph**.

## Why a graph, not a search

"Who calls this function?" is a graph question. Answer it with a regex — search for the name followed by `(` — and you get false hits on comments, strings, unrelated same-named symbols, and definitions, while missing calls that span files or are renamed on import.

ministr resolves references against the symbol graph instead, so `ministr_references` returns the *actual* callers, implementors, and importers — each pointing back to a real symbol, not a line that happened to match.

## Why it matters for an agent

Before changing shared code, an agent needs to know the blast radius. The reference graph turns "is anything still using this?" and "where is this trait implemented?" into one call with a precise answer — the difference between a safe edit and a guess.

References resolve across files, and connections that can't be settled in one pass are completed as indexing continues, so the graph is consistent rather than partial.
