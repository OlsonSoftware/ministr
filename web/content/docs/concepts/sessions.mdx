---
title: Sessions
description: How ministr tracks what each agent has received — dedup, delta delivery, compression, and honest accounting.
---

Context is the scarce resource in agent work. ministr keeps a per-session ledger of everything it has delivered, and uses it to never charge twice for the same content.

## Dedup and delta delivery

- **Search results already delivered are excluded** before truncation — a repeated `ministr_survey` returns new material, with a `deduplicated_count` noting what was skipped.
- **Re-reading an unchanged section** returns a short `already_delivered` stub instead of the full text.
- **Re-reading a changed section** is re-delivered as a delta update — the agent gets the new content precisely because it changed underneath.
- If an agent re-requests something it was assumed to still hold, ministr treats that as evidence the content was evicted from the agent's context and corrects its model.

The contract has an agent-side half: after dropping content from context, call [ministr_dropped](/docs/tools/dropped) so future reads return full text again. To keep something referenceable in less space, [ministr_compress](/docs/tools/compress) produces extractive summaries roughly 60–80% shorter.

## Coherence

When an indexed file changes after its content was delivered to a session, subsequent tool responses carry `coherence_alerts` naming the affected sections — the signal to re-read and pick up the delta. The desktop app's freshness display and this mechanism are two views of the same change tracking.

## Honest accounting

[ministr_usage](/docs/tools/usage) reports what ministr has delivered this session — deliveries, dedup hits, delta updates, compressions, and estimated token totals. Two honesty rules are designed in:

- The estimate is anchored to a **configured** window (`MINISTR_CONTEXT_WINDOW`), not the model's real one, and the tool's own description says so: it is advisory, and agents are told not to use it to conclude they are low on context.
- The server's instructions to agents deliberately never mention the budget machinery — early versions that surfaced it caused agents to wrongly abandon work.

## Session identity and recovery

Sessions are tracked per MCP connection (with deterministic IDs in stateless HTTP mode) and persisted, so delivery state survives daemon restarts. Activity events (which tool, when, cache hit, token delta) feed the desktop app's proof feed and the read→edit outcome receipts.
