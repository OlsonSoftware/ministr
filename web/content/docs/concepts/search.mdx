title: Search
description: What ministr_survey actually does — hybrid retrieval, multi-resolution ranking, optional reranking.
---

[ministr_survey](/docs/tools/survey) is not a text grep and not a plain vector lookup. A query runs through up to four stages:

## 1. Multi-resolution dense search

The query is embedded and searched against every vector resolution at once — document summaries, section summaries, full sections, claims, and code symbols. Candidates are scored by raw cosine similarity and **max-pooled per content item**: of all the resolutions a section matched at, only its best-scoring vector survives, so the top-k always holds distinct content. (This replaced an earlier hand-tuned per-resolution weight table — on deterministic evaluation the weights *lost* to plain max-pooling on both docs and code.) A question can be answered by a doc paragraph, a single claim, or a function, whichever actually matches.

## 2. Hybrid sparse fusion

With `sparse_weight > 0` in [configuration](/docs/guides/configuration#hybrid-retrieval-sparse_weight), ingestion also builds a sparse (SPLADE) keyword index and dense + sparse rankings are fused with weighted reciprocal-rank fusion. This is the exact-identifier recovery path for code: a search for a specific function name ranks it first even when embedding similarity alone would not. The weight is deliberately per-corpus — code corpora measured best at `0.6`, while documentation corpora measured best with fusion off entirely.

## 3. Matryoshka two-stage rescoring

If the corpus uses a truncated embedding dimension (see `dimension` in [configuration](/docs/guides/configuration)), coarse hits from the small-dimension index are re-scored against stored full-dimension vectors and blended — small index, full-precision ranking.

## 4. Cross-encoder reranking (optional)

With a `reranker_model` configured, retrieval over-fetches candidates and a cross-encoder rescores them against the query before truncation. Off by default.

## Session awareness

Results the agent has **already received are excluded server-side** (the response reports `deduplicated_count`), so repeated searching never wastes context on repeats. Strong hits come back with `next_actions` suggestions (read this section, open this symbol); weak, clustered results may trigger an elicitation asking for a refined query. See [sessions](/docs/concepts/sessions).

## Cross-corpus search

`corpus_ids` fans a query out across multiple corpora ([linked projects](/docs/guides/configuration#linked--multi-project-querying) or cloned repos), tags each hit with `source_corpus`, and merges by score. `corpus_boost` applies per-corpus multipliers (e.g. float your own repo above a dependency's). Fan-out is sequential in the current implementation.
