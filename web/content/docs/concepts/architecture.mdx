## The pieces

| Crate | Role |
|---|---|
| `ministr-core` | indexing pipeline, parsers, embedders, search, storage |
| `ministr-api` | shared types + the daemon HTTP client + the cloud seam trait |
| `ministr-daemon` | the shared background engine (one per machine) |
| `ministr-mcp` | the MCP server: 25 tools, sessions, instructions |
| `ministr-cli` | the `ministr` binary: serve, init, index, setup, … |
| `ministr-app` | the Tauri desktop app |

All six are MIT-licensed and build a complete local product with no cloud dependencies.

## One daemon, many clients

The interesting architectural fact: **`ministr serve` (stdio, the default) is a thin proxy.** The engine lives in a single shared daemon per machine, reached over a Unix domain socket at `~/.ministr/ministrd.sock` (a named pipe on Windows) — not a TCP port. There is no localhost HTTP server to secure; access control is filesystem permissions.

- The first client (MCP proxy or desktop app) **auto-spawns** the daemon as a detached process. It deliberately outlives its spawner: closing your editor or the app leaves indexes warm for the next session.
- The proxy serves the MCP handshake immediately; corpus registration and warm-up happen in the background, and the daemon lazy-loads indexes on first query.
- The daemon owns all state: per-corpus indexes under `~/.ministr/corpora/<id>/`, the run-history store, activity tracking, and the freshness cache. Corpus identity is a deterministic hash of the canonicalized path set, shared by every component — the CLI, daemon, and app can never disagree about which index a project maps to.
- The desktop app is **just another daemon client**, polling the same HTTP-over-UDS surface (see the [HTTP API](/docs/http-api)).

`ministr serve --transport http` is the opposite mode: a full in-process engine serving MCP over Streamable HTTP plus the REST surface — for remote and self-hosted deployments. See [self-hosting](/docs/operator/self-hosting).

## The open-core seam

Cloud features (orgs, SSO, billing, hosted corpora) attach through exactly one seam: a `CloudRouterMounter` trait in `ministr-api`, invoked once at HTTP-serve boot. The MIT binary always passes `None` — no cloud code paths exist in it at compile time. The official cloud binary wires in a private implementation. Self-hosting with no cloud env vars set is the fully-supported community mode, not a degraded one.

## Where data flows

```
your repo ──(index: hash, parse, embed)──▶ ~/.ministr/corpora/<id>/
                                              │  SQLite (source of truth)
                                              │  + HNSW vector index (derived cache)
agent ──MCP/stdio──▶ proxy ──UDS──▶ daemon ───┘
desktop app ─────────polls──UDS──▶ daemon
```

SQLite is the source of truth; the in-memory vector index is rebuilt from it whenever they could disagree. Nothing leaves your machine.
