title: Freshness
description: The no-mtime invariant — why every freshness verdict is hash-verified, and what the four states mean.
---

"Is my AI up to date with my code?" is the question ministr's UI exists to answer — so the answer is held to a strict standard:

> **Every freshness verdict shown to a user was produced by hashing the working tree.** File timestamps are never trusted for reporting. There is deliberately no stat/mtime shortcut on this path.

An editor that rewrites a file with an old mtime, a git checkout that restores timestamps, a build tool that touches without changing — none of these can make ministr claim "up to date" falsely or "behind" spuriously. (Internally, indexing uses mtime+size only as an *accelerator* to decide what to re-check; the per-file content hash always has the final word, and reporting never takes the shortcut at all.)

## The four states

| State | Meaning |
|---|---|
| `current` | the indexed content matches the file on disk, byte-for-byte by hash |
| `stale` | the file changed since indexing — your AI sees an older version |
| `new` | the file exists but has never been indexed |
| `missing` | the file was indexed but no longer exists — your AI still remembers it |

While a reindex runs, stale and new files are exactly what it's consuming, so the app shows them as **updating** (⟳) rather than alarming you about work already in progress.

## Cost control without weakening the contract

Hash-sweeping a tree isn't free, and the app polls every few seconds. The daemon dedupes: a process-wide cache holds each corpus's last verified sweep for **2.5 seconds** — below the fastest poll cadence — so overlapping polls collapse onto one sweep. A cached answer *is* a full hash-verified sweep, just seconds old. The cache is invalidated on reindex. The invariant is never traded for speed.

## Where you see it

- The app's trust panel and project mirror (see [the desktop app](/docs/desktop-app)) translate these states into plain words and the ✓ ⚠ ✗ ⟳ marks.
- The [HTTP API](/docs/http-api) exposes both the counts-only summary (`/freshness-summary`) and the full per-file sweep (`/freshness`).
- Agents feel it as `coherence_alerts` in tool responses: when a file changes after its content was delivered, the next response says so, and re-reading yields the delta.
