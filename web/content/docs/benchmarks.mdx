The landing page says ministr returns the slice that answers the question instead of the files it lives in. This page makes that concrete with a **reproducible synthetic micro-benchmark**, and tells you exactly how to run it.

## What this measures (and what it doesn't)

It isolates one thing: the **retrieval cost** — how many tokens cross the context window to answer a question — for a naive "find candidates, read every file" workflow versus a ministr-style targeted extract.

It is deliberately synthetic: one relevant function inside a codebase of otherwise-irrelevant, similarly-shaped files. That isolates the variable (how much irrelevant text the agent pays for) and makes the result deterministic. It is **not** an end-to-end measure of agent quality, latency, or answer correctness — only the token economics of retrieval.

## Result

Token counts use the same tokenizer ministr uses internally. The targeted extract is a constant 68 tokens; the naive workflow grows with the number of candidate files it reads.

| candidate files | grep + read tokens | ministr tokens | reduction |
|---|---|---|---|
| 5   | 1,215  | 68 | 94.4% |
| 20  | 4,860  | 68 | 98.6% |
| 50  | 12,150 | 68 | 99.4% |
| 100 | 24,300 | 68 | 99.7% |

The point isn't the exact percentage — it's the shape: the naive cost scales with how much irrelevant code the agent has to read to be sure, while the targeted cost stays flat. On a real codebase the relevant slice is rarely the whole file, so the gap is directionally the same.

## Reproduce it

The figures above are produced by a deterministic test (no network, no model download):

```sh
cargo test -p ministr-core --test token_economics_report -- --nocapture
```

It prints the table and asserts the relationship holds (reduction increases with codebase size; ≥ 90% at 100 files), so the published claim can't silently drift.

A companion **latency** benchmark (Criterion) times the tokenizer and retrieval-shaping path rather than counting tokens:

```sh
cargo bench -p ministr-core --bench token_economics
```

## Honest caveats

- Synthetic inputs. Real files vary; treat the percentages as illustrative of the *shape*, not a guarantee for your repo.
- Retrieval cost only. It doesn't credit ministr for code-intelligence quality (symbols, references, cross-language bridges) or charge it for index build time.
- Single-turn. Session efficiency (not re-sending what the agent already has) compounds this over a conversation but is measured separately.
