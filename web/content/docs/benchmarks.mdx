The landing page says ministr returns the slice that answers the question instead of the files it lives in. This page makes that concrete with a **real end-to-end measurement** — actual indexing, actual retrieval, actual response bytes — and tells you exactly how to run it.

## What this measures (and what it doesn't)

It isolates one thing: the **retrieval cost** — how many tokens cross the context window to answer a question — for a naive "find candidates, read every file" workflow versus a real `ministr_survey` lookup.

It is **not** an end-to-end measure of agent quality, latency, or answer correctness — only the token economics of retrieval. Token counts use the same `cl100k` tokenizer ministr meters with internally.

## Result — real measurement

The benchmark indexes a real, committed multi-language corpus (`eval/corpus-code`: Rust, Python, Go, TypeScript, Java, C++ — 6 files, ~3,100 tokens) through ministr's real ingestion pipeline, then for each of 26 real natural-language queries it measures:

- **ministr** — the *literal bytes* a `ministr_survey` call returns through the MCP tool interface (ranked top-k slices plus the condensed envelope). This is exactly what the agent receives, not a hand-picked extract.
- **grep + read** — every corpus file that shares a salient term with the query, read whole (the "grep for the terms, then read each candidate file to be sure" workflow), over the same real files.

| | grep + read | ministr | reduction |
|---|---|---|---|
| **Aggregate (26 queries)** | 40,256 tokens | 13,477 tokens | **66.5% fewer** |
| **Mean per lookup** | 1,548 tokens | 518 tokens | 66.5% fewer |

A ministr lookup is **not** a fixed cost — a survey returns the top-k ranked slices, averaging ~518 tokens here. It came out cheaper on **23 of 26** queries. On the other 3, the query matched a single small file, so reading that one file beat a 5-result survey. That's the honest shape: the savings are real and grow with how much irrelevant code grep would otherwise drag in, but a targeted lookup is bounded, not free.

## How the gap scales (illustrative model)

The measured corpus is small. The reason the advantage *widens* on real codebases is structural: a targeted lookup's cost is bounded by the result budget regardless of repo size, while "read every candidate file" grows with the number of files a keyword surfaces. A deterministic synthetic model isolates that scaling — one relevant function in a codebase of otherwise-irrelevant, similarly-shaped files:

| candidate files | grep + read tokens | ministr (one extract) | reduction |
|---|---|---|---|
| 5   | 1,215  | 68 | 94.4% |
| 20  | 4,860  | 68 | 98.6% |
| 50  | 12,150 | 68 | 99.4% |
| 100 | 24,300 | 68 | 99.7% |

These are an **illustrative upper bound**, not a measurement: they assume the relevant slice is a single small extract and every candidate file is read whole. The real measurement above is the honest figure; this model just shows why the gap grows with scale.

## Reproduce it

Both are deterministic (no network, no model download — a hash embedder keeps token *volume*, which is what's measured, independent of embedding quality):

```sh
# Real end-to-end measurement (the table above): index a real corpus,
# run real ministr_survey calls, count the literal response tokens.
cargo test -p ministr-mcp --test token_economics_e2e report_real_token_economics -- --nocapture

# Illustrative scaling model.
cargo test -p ministr-core --test token_economics_report -- --nocapture
```

The e2e test also asserts the relationship holds (a survey stays well under the token budget; aggregate grep+read exceeds aggregate ministr; ministr wins a majority of queries), so the published claim can't silently drift.

A companion **latency** benchmark (Criterion) times the tokenizer and retrieval-shaping path rather than counting tokens:

```sh
cargo bench -p ministr-core --bench token_economics
```

## End-to-end agent benchmark (does it help the agent?)

Retrieval-token cost is a proxy. The stronger question is whether a real coding
agent, given ministr, reaches a **correct** solution with less work. A second
harness (`benchmarks/agent-task/`) measures exactly that: it runs the *same*
headless coding agent on the *same* task twice, changing only the discovery tool
— ministr vs `grep`/`glob` — across a difficulty ladder of fixtures and real
open-source repos (with the repo's own failing→passing tests as a deterministic
validator), and across models.

Measured so far (`2026-06-05`, headless Claude Code). **These are early,
small-sample numbers — read the trial counts and treat them as directional, not
a headline.** Correctness was a tie everywhere (both arms solved every task), so
the comparison is **cost** at equal correctness:

| task (repo size) | the task | model | trials | both solved? | ministr | grep | head-to-head |
|---|---|---|:--:|:--:|--:|--:|--|
| sympy (~1,590 files) | symptom only, no test named | **sonnet** | 2 | ✅ | **$0.198** | $0.334 | **ministr 41% cheaper** |
| sympy (~1,590 files) | symptom only, no test named | haiku | 2 | ✅ | $0.260 | $0.253 | ~even |
| click (~70 files) | names the failing test | sonnet | 1 | ✅ | $0.083 | $0.082 | ~even |
| minicsv (9 files, synthetic) | names the failing test | sonnet | 1 | ✅ | $0.090 | $0.082 | ~even |

The pattern, stated honestly:

- **Both levers have to be present.** ministr's cost edge shows up only when the
  codebase is **large** *and* the task forces **discovery** (a symptom with no
  pointer to the failing test/function). On the huge sympy repo with a
  symptom-only task, a capable model with ministr found the fix using ~half the
  search-related tokens and fewer turns. On small repos, or when the task names
  the test, `grep` ties — there's nothing to discover.
- **A weaker model benefits less** (haiku ~even) — it spends turns scanning
  regardless, so a better search tool helps it less.
- **Sample sizes are tiny** (1–2 trials/cell). The sympy/sonnet effect is large
  and directionally consistent, but this is an early signal, not a published
  guarantee. More trials and more repos are the obvious next step.

Run it yourself (it calls a real model, so it is opt-in):

```sh
python3 benchmarks/agent-task/run_sxs.py --selftest          # no-LLM: tasks are valid red→green
python3 benchmarks/agent-task/run_sxs.py --models haiku      # the matrix
```

## Honest caveats

- Retrieval cost only. It doesn't credit ministr for code-intelligence quality (symbols, references, cross-language bridges) or charge it for index build time.
- A lookup is bounded, not flat. The per-call cost varies with `top_k` and how much each result carries; on a tiny corpus a broad survey can occasionally cost as much as a tight grep.
- Single-turn. Session efficiency (not re-sending what the agent already has) compounds this over a conversation but is measured separately.
