---
title: SAML via an OIDC bridge
description: The supported path for SAML-only IdPs today — bridge to OIDC; native assertion consumption is not yet shipped.
---

**Honest status first:** ministr ships SAML SP *metadata* and *login initiation* (`GET /orgs/{id}/saml/metadata.xml`, `GET /orgs/{id}/saml/login`), but the assertion-consumer endpoint — the part that actually consumes a signed SAML response — is **not implemented yet**. Per-org SAML config (IdP entity ID, SSO URL, x509 cert, attribute mappings, signed-assertion enforcement) exists and is CRUD-able, ready for when the ACS lands.

## The supported path: bridge SAML to OIDC

If your IdP is SAML-only (or your org mandates SAML federation), put an identity bridge in front and configure ministr's [OIDC](/docs/operator/oidc) against the bridge:

```
ministr  ──OIDC──▶  bridge (Auth0 / Okta / Keycloak / Azure AD)  ──SAML──▶  your IdP
```

1. Configure the bridge as a SAML **service provider** against your IdP (the bridge's docs cover this; ministr's `/saml/metadata.xml` is not involved — the bridge presents its own SP metadata).
2. Create an OIDC application on the bridge.
3. Point ministr's org OIDC config at the bridge (`issuer_url`, `client_id`, `client_secret`), and map your IdP's group attribute through the bridge into the `groups_claim`.

From ministr's perspective this is plain OIDC; everything on the [OIDC page](/docs/operator/oidc) applies, including the in-memory pending-login caveat and bearer-token session model.

## Why native SAML is pending

Native assertion validation depends on an XML-signature stack whose current Rust bindings have platform-specific issues; rather than ship a partially-verified ACS, ministr ships none. The metadata and login-initiation routes exist so SP↔IdP exchange can be staged ahead of the cutover. When the native path lands, configured orgs switch without re-federation.
