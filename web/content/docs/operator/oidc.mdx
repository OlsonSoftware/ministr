OIDC is the supported enterprise SSO path (for SAML-only IdPs, see [SAML](/docs/operator/saml)). The implementation is pure Rust (openidconnect + rustls + jsonwebtoken) — no native XML/SSL libraries involved.

## Configure an org

`POST /api/v1/orgs/{id}/oidc/config` (owner/admin only, scope `ministr:read` for the management routes):

| Field | Required | Default | Meaning |
|---|---|---|---|
| `issuer_url` | yes | — | the IdP's OIDC issuer (discovery is fetched from here, cached ~1h) |
| `client_id` | yes | — | OAuth client registered at the IdP |
| `client_secret` | yes | — | echoed back only as `[REDACTED]` |
| `groups_claim` | no | `groups` | claim carrying group memberships |
| `email_claim` | no | `email` | claim carrying the user's email |
| `name_claim` | no | `name` | display-name claim |
| `enforce_email_verified` | no | `true` | reject ID tokens without `email_verified` |
| `group_role_map` | no | `{}` | IdP group → ministr role; empty means no role inference |

`GET` returns the config (secret redacted); `DELETE` removes it (204). No config row means SSO is simply not enabled for that org (404 on the login route).

Register this redirect URI at your IdP:

```
{MINISTR_CLOUD_BASE_URL}/orgs/{org_id}/oidc/callback
```

## The flow

1. Users start at `GET /orgs/{id}/oidc/login` — ministr fetches the IdP's discovery document, builds an authorize URL with PKCE (S256), nonce, and state, and redirects.
2. The callback exchanges the code, validates the ID token (JWKS signature, issuer, audience, nonce, and `email_verified` if enforced), and upserts the user by email.
3. The response is a bearer token (`{token, user_id, plan_id}`) minted through the same store as GitHub sign-in — downstream, a federated user is indistinguishable. Default scope on SSO sign-in: `ministr:read ministr:write`.

## Operational limits

- Pending login state (state/nonce/PKCE verifier) is held **in memory**: a server restart between redirect and callback invalidates in-flight logins; users just retry.
- The IdP discovery document is cached for about an hour — IdP config changes can take that long to be seen.
- For long-lived programmatic access use API keys (`mst_pk_…`, individually revocable) rather than minting OAuth tokens.
