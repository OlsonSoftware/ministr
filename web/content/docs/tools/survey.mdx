`ministr_survey` searches the indexed corpus with a natural-language query. Start here for any vague question your agent has about the codebase or docs, then follow up with `ministr_read` (full text) or `ministr_extract` (atomic claims) on the top results.

## Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `corpus_boost` | object | no | Optional per-corpus score multipliers for cross-corpus ranking. Map\<corpus_id, multiplier\>; absent corpora default to 1.0; clamped to [0, 10]. Use 2.0 to float your own repo above Atlas hits, 0.0 to suppress a corpus. |
| `corpus_ids` | array of string | no | Optional cross-corpus list. When set and non-empty, fans the query out across each corpus_id (own corpora or Atlas slugs), tags hits with source_corpus, and merges results by score. Omit to query a single corpus. |
| `project` | string | no | Optional linked-project label (from .ministr.toml [[linked]]). Omit for the session's primary corpus. Call ministr_projects to list labels. |
| `query` | string | no | Natural language query to search the corpus |
| `top_k` | integer | no | Maximum number of results to return |

Annotations: read-only · idempotent.

## Returns

A `SurveyResponse` with `results` and `deduplicated_count`. Each result carries `content_id`, `resolution` (document, section, claim, or symbol), `score`, `text`, `heading_path`, and `source_corpus`.

## Notes

- Results your agent has already received are excluded server-side before truncation, so repeat surveys don't re-deliver the same content.
- If the results are low-score and clustered, the tool may issue an elicitation asking the client for a `refined_query`. This is skipped on the cross-corpus path.
- A strong top hit emits a `next_actions` suggestion: `ministr_read` for sections, `ministr_definition` for symbols.
- A survey warms a prefetch cache of parent sections (local mode only), making a follow-up `ministr_read` cheaper.
- The response is wrapped in the standard envelope: `coherence_alerts`, `indexing_in_progress`/`indexing_message`, and `next_actions`. Logical failures return soft errors (`ok: false`, `error_kind`, `message`) and never cancel sibling parallel tool calls.
